# UI/UX Improvement Plan — Amplitude Frontend

> Статус: **Ожидает подтверждения**  
> Автор анализа: GitHub Copilot  
> Дата: 2026-05-04

---

## Контекст

Фронтенд — Next.js 15 + TypeScript + Tailwind CSS v4. Без UI-библиотеки.  
Все компоненты написаны вручную. Авторизация через `localStorage` + Token.  
Основные страницы: Analytics, Bonus Transactions, Coupon Dispatch, Push Dispatch, Blacklist.

---

## БЛОК 1 — Критические баги / Сломанная функциональность

### 1.1 Blacklist страница полностью не работает
**Проблема:** Страница `/blacklist` имеет форму, но:
- нет обработчика `onSubmit`
- нет API вызовов
- нет таблицы с данными
- отображается заглушка-текст вместо реального контента

**Решение:**  
Подключить API endpoints для blacklist, реализовать CRUD таблицу и форму добавления.

---

### 1.2 `TopNavbar` — мёртвый код
**Проблема:** Компонент `TopNavbar` полностью написан, но нигде не рендерится — был вытеснен `AppShell`. Занимает место в бандле, вводит в заблуждение.

**Решение:** Удалить файл `features/analytics/components/TopNavbar.tsx`.

---

### 1.3 Race condition в поиске marketing sales (Coupon Dispatch)
**Проблема:** Debounce через `setTimeout` + `AbortController`, но `abort signal` никогда не передаётся в `listMarketingSales` — при быстром наборе запросы гонятся.

**Решение:** Передать `AbortSignal` в fetch-запрос к `/api/coupon-dispatch/marketing-sales/`.

---

## БЛОК 2 — Загрузочные состояния (Loading States)

### 2.1 Auth check — нет полноэкранного лоадера
**Проблема:** Пока `useAuthGuard` проверяет токен, показывается только центрированный текст `"Проверка доступа..."` на белом фоне — выглядит как зависший экран.

**Решение:** Полноэкранный спиннер с логотипом и анимацией в стиле дизайна.

---

### 2.2 StatsCards — дефисы вместо скелетонов
**Проблема:** Карточки статистики показывают `"—"` при загрузке — выглядит как "нет данных", а не "грузится".

**Решение:** Skeleton-плейсхолдеры (анимированный pulse) для каждой карточки во время загрузки.

---

### 2.3 ActivityTable — нет skeleton rows
**Проблема:** При загрузке таблица пуста — пользователь не понимает, идёт ли загрузка или данных нет.

**Решение:** 8–10 skeleton строк с анимацией во время `loading=true`.

---

### 2.4 UserHistoryPanel — нет индикатора загрузки истории
**Проблема:** При клике "Загрузить историю" нет spinner/skeleton — кнопка просто становится неактивной.

**Решение:** Показывать skeleton timeline пока идёт запрос.

---

### 2.5 Coupon/Bonus job detail — нет лоадера при polling
**Проблема:** При polling каждые 8 секунд нет визуального подтверждения что данные обновляются.

**Решение:** Небольшой `RefreshCw` иконка-спиннер в углу карточки задания (не блокирующий).

---

### 2.6 Push Dispatch — нет индикатора загрузки городов
**Проблема:** При переключении на target "city" список городов грузится без индикации.

**Решение:** Skeleton для select-box городов.

---

## БЛОК 3 — Обработка ошибок (Error Handling)

### 3.1 Analytics: `loadUserHistory` — silent errors
**Проблема:** В `finally` блоке нет `catch` — ошибки при загрузке истории пользователя полностью игнорируются.

**Решение:** Добавить `catch` и показывать inline-ошибку в `UserHistoryPanel`.

---

### 3.2 Push Dispatch — stale error state
**Проблема:** `error` state используется и для ошибки загрузки городов, и для ошибки отправки. При повторной отправке старая ошибка может оставаться видимой.

**Решение:** Разделить на `citiesError` и `sendError` — независимые состояния.

---

### 3.3 Polling (Coupon/Bonus) — silent fetch errors
**Проблема:** `catch {}` при polling проглатывает все ошибки — если бэкенд вернул 500, пользователь не узнает.

**Решение:** После N неудачных попыток показать inline-предупреждение "Не удалось обновить данные".

---

### 3.4 Нет глобального обработчика непредвиденных ошибок
**Проблема:** Если API возвращает неожиданный формат, приложение может крашнуться молча.

**Решение:** Добавить Error Boundary на уровне layout с дружелюбным сообщением об ошибке.

---

## БЛОК 4 — Пустые состояния (Empty States)

### 4.1 ActivityTable — нет разграничения "грузится" / "нет данных"
**Проблема:** `"Нет данных"` показывается и при `loading=true` с предыдущим пустым результатом — флеш пустого состояния при каждом рефреше.

**Решение:** Показывать `"Нет данных"` только когда `loading=false && rows.length === 0`.

---

### 4.2 Job history (Bonus/Coupon) — нет иллюстрации для пустого списка
**Проблема:** При пустом списке заданий — просто ничего нет.

**Решение:** Иллюстрация + текст "Заданий пока нет. Создайте первое задание."

---

### 4.3 Push Dispatch — нет подтверждения успеха
**Проблема:** После успешной отправки пуша нет явного визуального успеха (только текст меняется).

**Решение:** Toast/banner с зелёным фоном и иконкой ✓ на 5 секунд.

---

## БЛОК 5 — Дизайн-система и Визуальная Иерархия

### 5.1 Нет унифицированных UI-компонентов
**Проблема:** `Button`, `Input`, `Select`, `Badge`, `Modal` — каждый пишется заново inline на каждой странице. Визуальные несоответствия.

**Решение:** Создать `features/common/components/ui/`:
- `Button.tsx` — с вариантами `primary`, `secondary`, `danger`, `ghost` и состоянием `loading`
- `Input.tsx` — с поддержкой `label`, `error`, `helpText`
- `Textarea.tsx`
- `Select.tsx`
- `Badge.tsx` — статусные метки (`pending`, `processing`, `completed`, `failed`)
- `Spinner.tsx` / `Skeleton.tsx`
- `Modal.tsx` — с управлением scroll lock
- `Toast.tsx` — уведомления (success/error/info)
- `Card.tsx` — унифицированная карточка

---

### 5.2 Badge статусов заданий — нет иконок и консистентности
**Проблема:** Статусы `pending/processing/completed/failed` отображаются просто текстом или с минимальным цветом.

**Решение:**
- `pending` — серый, иконка часы
- `processing` — синий, анимированный спиннер
- `completed` — зелёный, иконка ✓
- `failed` — красный, иконка ✗

---

### 5.3 Нет Toast-системы
**Проблема:** Успех/ошибка показываются через статичные div-баннеры в контенте страницы — теряются при скролле.

**Решение:** Глобальная toast-система (fixed позиция, top-right), исчезает через 5 сек, поддерживает очередь сообщений.

---

### 5.4 Кнопки без состояния `loading`
**Проблема:** При нажатии "Отправить" / "Создать задание" кнопка не показывает загрузку — можно кликнуть дважды.

**Решение:** Кнопки показывают inline-спиннер и блокируются при `isLoading=true`.

---

### 5.5 Монотонная палитра — нет визуальных акцентов
**Проблема:** Весь UI в оттенках синего/серого. Трудно выделить важное.

**Решение:** Определить цветовую систему:
- `primary`: indigo-600 (текущий, оставить)
- `success`: emerald-600
- `warning`: amber-500  
- `danger`: rose-600
- `neutral`: slate-серые

---

### 5.6 Мобильная адаптация — неполная
**Проблема:** Таблицы (ActivityTable, job results) не адаптированы для мобильных — горизонтальный скролл без индикации, мелкий текст.

**Решение:** Card-view на мобильных для всех таблиц (под `md:` breakpoint).

---

## БЛОК 6 — Компонентная архитектура

### 6.1 `CalendarField` дублируется
**Проблема:** Почти идентичная логика выбора даты написана и в `FiltersBar` (analytics) и inline на странице bonus-transactions (~200 строк). 

**Решение:** Вынести в `features/common/components/CalendarField.tsx` — один переиспользуемый компонент.

---

### 6.2 Логика создания задания в компоненте страницы
**Проблема:** Страницы `/bonus-transactions` и `/coupon-dispatch` содержат >500 строк каждая — форма, список, детальный вид всё в одном файле.

**Решение:** Разбить на компоненты:
- `CreateJobForm.tsx`
- `JobList.tsx`
- `JobDetail.tsx`

---

### 6.3 Нет useJobPolling хука
**Проблема:** Идентичный polling-код (setInterval + visibilitychange + cleanup) скопирован в `/bonus-transactions` и `/coupon-dispatch`.

**Решение:** Вынести в `features/common/hooks/useJobPolling.ts`.

---

## БЛОК 7 — Доступность (Accessibility)

### 7.1 Нет `aria-live` для уведомлений
**Проблема:** Screen readers не анонсируют сообщения об успехе/ошибке.

**Решение:** Обернуть toast/banner в `<div aria-live="polite">`.

---

### 7.2 Модальное окно (Push Dispatch) — нет scroll lock
**Проблема:** При открытии модала подтверждения страница под ним скроллится.

**Решение:** `document.body.style.overflow = 'hidden'` при открытии, восстановление при закрытии.

---

### 7.3 Нет `focus trap` в модалах
**Проблема:** Tab навигация уходит из модала.

**Решение:** Реализовать focus trap в `Modal.tsx`.

---

## БЛОК 8 — Безопасность

### 8.1 Auth token в `localStorage` (XSS-уязвимость)
**Проблема:** `localStorage` доступен через XSS-атаки.

**Оценка:** Риск приемлем если нет XSS-векторов, но лучшая практика — HttpOnly cookie.

**Решение (опционально):** Добавить endpoint `/api/auth/set-cookie/` на Next.js стороне, который устанавливает HttpOnly cookie, и читать токен оттуда. (Требует координации с бэкендом.)

## Итог

**Всего проблем выявлено: 30+**

Самые критичные для UX:
1. Отсутствие loading states — пользователь не понимает что происходит
2. Молчаливые ошибки — пользователь не знает что что-то сломалось
3. Кнопки без блокировки — возможны дублирующие запросы
4. Нет единого UI kit — каждая страница выглядит чуть по-разному

После реализации фронтенд станет production-ready с предсказуемыми состояниями, понятным feedback-ом и консистентным дизайном.
