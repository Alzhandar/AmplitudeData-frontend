# Аналитика регистраций: требования системного аналитика

## 1. Контекст и цель

Блок аналитики регистраций предназначен для руководителей и лидов, которые должны
принимать решения на основе данных: работают ли маркетинговые кампании, растёт ли
база, нет ли аномальных провалов.

Ключевой вопрос, на который должен отвечать блок:

> **Сколько новых пользователей пришло за выбранный период — и это много или мало
> относительно нормы?**

---

## 2. Аудитория и сценарии использования

| Роль          | Типичный вопрос                                                       |
|---------------|-----------------------------------------------------------------------|
| Руководитель  | Сколько зарегистрировалось за последний месяц? Растёт ли база?        |
| Маркетолог    | Дала ли кампания прирост? Был ли скачок в конкретный день?            |
| Аналитик      | Какой средний дневной прирост? Как этот месяц против прошлого?        |
| Разработчик   | Нет ли аномального падения регистраций после релиза?                  |

---

## 3. Критические метрики (что должно быть в блоке)

### 3.1 Первичные (без них блок бессмысленен)

| Метрика                   | Описание                                              | Источник данных      |
|---------------------------|-------------------------------------------------------|----------------------|
| **Новых за период**       | Абсолютное число регистраций за выбранный диапазон    | `registrations`      |
| **Изменение к периоду**   | Delta vs. предыдущий аналогичный период (+N / −N / %) | 2 запроса к API      |
| **Всего в базе**          | Накопленное число пользователей на конец периода      | `total_users`        |
| **Доля прироста**         | `registrations / total_users × 100 %`                 | вычисляется          |

### 3.2 Вторичные (дают контекст)

| Метрика                   | Описание                                              | Источник данных      |
|---------------------------|-------------------------------------------------------|----------------------|
| **Средний в день**        | `registrations / кол-во дней в периоде`               | вычисляется          |
| **Дневной тренд**         | Столбчатый мини-график по дням                        | нужен новый endpoint |
| **Пиковый день**          | День с максимальными регистрациями                    | нужен новый endpoint |

---

## 4. Что значит «изменение к предыдущему периоду»

Если выбран период `01.04 — 30.04` (30 дней), то система автоматически сравнивает
его с `01.03 — 31.03` (предыдущие 30 дней) и показывает:

```
+312   ↑ 18%   относительно предыдущего периода
```

Это **самая важная** метрика для руководителя: не само число, а тренд.

---

## 5. Обязательные взаимодействия

| Взаимодействие              | Зачем                                                      |
|-----------------------------|------------------------------------------------------------|
| Быстрые пресеты периода     | 7д / 30д / 90д / С начала года — самые частые запросы      |
| Ручной выбор дат            | Произвольный анализ, проверка кампаний                     |
| Автообновление при смене    | Данные должны загружаться сразу после выбора периода       |
| Skeleton при загрузке       | Нет "прыжков" вёрстки, понятно что данные грузятся         |
| Корректная ошибка           | Если API недоступен — понятное сообщение, не белый экран   |

## 6. Визуальная иерархия информации

```
┌─────────────────────────────────────────────────────┐
│ [иконка]  Регистрации в приложении    [7д][30д][90д] │
├─────────────────────────────────────────────────────┤
│                                                     │
│   1 234            +18%↑           avg 41/день      │
│   новых        vs пред. период                      │
│   1 апр — 30 апр                                    │
│                                                     │
│ ▁▂▃▄▄▅▅▃▂▄▅▆▇▆▅▄▅▄▅▆▇▇▆▅▄▅▃▄▅▃ (дневной спарклайн) │
├─────────────────────────────────────────────────────┤
│  Всего в базе: 58 421                               │
└─────────────────────────────────────────────────────┘
```

Иерархия восприятия:
1. Большое число (что произошло)
2. Дельта к прошлому периоду (хорошо или плохо)
3. Средний в день (темп)
4. График (где пики)
5. Всего в базе (масштаб)

## 7. Ограничения текущего API

Текущий бэкенд (`/api/amplitude/mobile-registrations-stats/`) возвращает:

```json
{
  "registrations": 1234,
  "total_users": 58421,
  "date_from": "2026-04-01",
  "date_to": "2026-04-30",
  "source": "mobile_api",
  "cached": false
}
```

**Что уже есть:**
- Число новых за период ✓
- Всего в базе ✓
- Диапазон дат ✓

**Чего не хватает:**
- Разбивка по дням (для графика) — нужен новый endpoint
- Данные за предыдущий период — нужен второй параллельный запрос (тот же endpoint, другие даты)
- Разбивка по платформе (iOS / Android) — нужна поддержка в upstream API

## 8. Приоритизация (MoSCoW)

| Что                                      | Приоритет   |
|------------------------------------------|-------------|
| Число новых + всего в базе               | Must have   |
| Дельта к предыдущему периоду             | Must have   |
| Средний в день                           | Must have   |
| Быстрые пресеты периода                  | Must have   |
| Ручной выбор дат                         | Should have |
| Дневной мини-график (спарклайн)          | Should have |
| Разбивка iOS / Android                   | Could have  |
| Экспорт в CSV                            | Won't have  |
