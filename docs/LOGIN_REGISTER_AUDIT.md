# Аудит страниц Login и Register

---

## РАЗДЕЛ A — Логика и код

### A1. Ошибки API без локализации ⚠️ Высокий

**Проблема:**
```ts
const message = err instanceof Error ? err.message : "Ошибка авторизации";
setError(message);
```
`translateErrorMessage()` уже существует в проекте и используется на всех других страницах. Здесь не применяется — пользователь может увидеть сырую английскую строку из API.

**Правка:** Обернуть `message` в `translateErrorMessage(message)`.

---

### A2. `localStorage` ключи — дублирующиеся литералы ⚠️ Высокий

**Проблема:** В обоих файлах:
```ts
window.localStorage.setItem("auth_token", result.token);
window.localStorage.setItem("auth_iin", result.iin);
```
Строковые ключи `"auth_token"` / `"auth_iin"` также встречаются в `use-auth-guard.ts`. Опечатка — незаметный баг.

**Правка:** Вынести в `features/auth/storage.ts`:
```ts
export const AUTH_TOKEN_KEY = "auth_token";
export const AUTH_IIN_KEY = "auth_iin";
export function saveAuthSession(token: string, iin: string) { ... }
export function clearAuthSession() { ... }
```

---

### A3. Поля не блокируются во время запроса ⚠️ Средний

**Проблема:** При `loading=true` только кнопка `disabled`, поля email/password остаются редактируемыми. Пользователь может изменить данные пока запрос летит — это race condition.

**Правка:** Добавить `disabled={loading}` на все `<input>`.

---

### A4. Нет атрибутов `autocomplete` ⚠️ Высокий

**Проблема:** Браузер и менеджеры паролей (1Password, keychain) не работают без `autocomplete`.

**Правка:**
- Login: `autocomplete="email"` + `autocomplete="current-password"`
- Register: `autocomplete="email"` + `autocomplete="new-password"` + `autocomplete="off"` (ИИН)

---

### A5. Поле ИИН не валидируется на клиенте ⚠️ Средний

**Проблема:** ИИН — строго 12 цифр. Принимается любая строка, пользователь узнаёт об ошибке только после запроса к серверу.

**Правка:**
- `inputMode="numeric"` — цифровая клавиатура на мобильных
- `maxLength={12}`
- `pattern="[0-9]{12}"`
- Hint под полем: `"12 цифр"`

---

## РАЗДЕЛ B — Визуальный интерфейс

### B1. Цвет `blue-600` вместо `indigo-600` — рассинхрон дизайна ⚠️ Высокий

**Проблема:** Кнопки и focus-состояния используют `blue-600` / `focus:border-blue-500`, весь остальной интерфейс — `indigo-600` / `focus:ring-indigo-400`. Страница входа визуально "из другого приложения".

**Правка:** Заменить `blue-*` → `indigo-*`. Кнопка — переиспользовать готовый `<Button variant="primary" size="lg">` из UI-кита (у него уже есть spinner, focus ring, disabled).

---

### B2. Нет логотипа / бренда ⚠️ Средний

**Проблема:** Страница входа — первое что видит пользователь. Белая карточка без какой-либо идентичности. `public/logo-new.svg` существует и используется в сайдбаре, но на auth-страницах его нет.

**Правка:** Добавить логотип над заголовком карточки.

---

### B3. Нет split-layout (левая панель + правая форма) ⚠️ Средний

**Проблема:** Центрированная карточка на сером фоне — минимальный уровень. Для B2B-инструмента стандарт (Notion, Linear, Vercel, Figma) — двухколоночный layout на десктопе:
- **Левая панель:** фирменный цвет `indigo-700`, логотип, короткий слоган, возможно иллюстрация
- **Правая панель:** форма на белом фоне

На мобильных — скрыть левую панель, показать только форму.

**Правка:** Обернуть в `grid grid-cols-1 lg:grid-cols-2`, левая колонка — `bg-indigo-700`.

---

### B4. Нет show/hide пароля ⚠️ Высокий

**Проблема:** Пользователь не может проверить что ввёл в поле пароля. Особенно критично при регистрации — он мог ошибиться и не узнает об этом.

**Правка:** Добавить кнопку-глаз внутри поля (positioned absolute right), переключать `type="password"` / `type="text"`.

---

### B5. Поля без `focus:ring` — размытый визуальный feedback ⚠️ Средний

**Проблема:**
```
focus:border-blue-500
```
Без `focus:ring` смена только цвета рамки почти незаметна. В остальном приложении используется `focus:ring-1 focus:ring-indigo-400 focus:border-indigo-400`.

**Правка:** Унифицировать: `focus:border-indigo-400 focus:outline-none focus:ring-2 focus:ring-indigo-400/30`.

---

### B6. Нет inline-валидации полей (feedback до submit) ⚠️ Средний

**Проблема:** Пользователь вводит email, уходит в следующее поле — никакого feedback. Узнаёт об ошибке только после нажатия кнопки. Это замедляет работу.

**Правка:** Состояние `touched` для каждого поля. При `onBlur` + невалидном значении показывать под полем текст ошибки красным:
- Email: "Введите корректный email"
- Пароль (при регистрации): "Минимум 8 символов"
- ИИН: "Должно быть ровно 12 цифр"

---

### B7. Нет индикатора надёжности пароля при регистрации ⚠️ Низкий

**Проблема:** Пользователь не знает насколько надёжен его пароль при регистрации.

**Правка:** Простой прогресс-бар под полем пароля (только на register):
- Слабый: красный (< 8 символов)
- Средний: оранжевый (≥ 8, нет цифр/спецсимволов)
- Сильный: зелёный (≥ 8 + цифры или спецсимволы)

---

### B8. Ошибка отображается без иконки — не хватает визуального веса ⚠️ Низкий

**Проблема:** Блок ошибки — просто текст в розовом прямоугольнике. Без иконки ⚠ или ✕ он теряется на фоне.

**Правка:**
```tsx
<div className="flex items-start gap-2 rounded-lg border border-red-200 bg-red-50 px-3 py-2">
  <svg ...>⚠</svg>
  <p className="text-sm text-red-700">{error}</p>
</div>
```

---

### B9. Нет анимации появления карточки ⚠️ Низкий

**Проблема:** Карточка появляется мгновенно — ощущение "жёсткой" загрузки. Небольшой fade+slide-up создаёт ощущение полированного продукта.

**Правка:** Tailwind `animate-in fade-in slide-in-from-bottom-4 duration-300` (если подключён `tailwindcss-animate`) или простой CSS-класс с `@keyframes`.

---

### B10. `autoFocus` не установлен ⚠️ Низкий

**Проблема:** При открытии страницы фокус не на первом поле. Пользователь обязан кликнуть, не может сразу начать вводить с клавиатуры.

**Правка:** `autoFocus` на поле email в обоих файлах.

---

## РАЗДЕЛ C — Доступность (a11y)

### C1. Нет `aria-live` на блоке ошибки ⚠️ Средний

**Проблема:** Скринридеры не объявляют ошибку когда она появляется — блок просто отрисовывается в DOM без уведомления.

**Правка:**
```tsx
<div role="alert" aria-live="assertive">
  {error && <p>...</p>}
</div>
```
Даже пустой `role="alert"` div можно вставить заранее — скринридер объявит содержимое при его добавлении.

---

### C2. Поля не связаны с ошибками через `aria-describedby` ⚠️ Низкий

**Проблема:** При наличии inline-ошибки поле не сообщает ассистивным технологиям что ошибка к нему относится.

**Правка:** `aria-describedby="email-error"` на `<input>` + `id="email-error"` на теге ошибки.

---

## РАЗДЕЛ D — Дублирование кода

### D1. Разметка карточки / стили полей дублируются ⚠️ Низкий

**Правка:** Вынести `<AuthFormField label hint inputProps>` в `features/auth/components/AuthFormField.tsx`. Используется в обоих файлах.
